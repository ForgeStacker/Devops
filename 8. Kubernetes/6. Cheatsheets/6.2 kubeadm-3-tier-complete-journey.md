# Kubeadm 3-Tier Application Deployment – Complete Learning Journey

This document records a full, hands-on lab: what I actually did in my kubeadm environment, including mistakes, troubleshooting, fixes and the Kubernetes concepts I learned.

## Goal

Deploy a complete 3‑tier application on Kubernetes using kubeadm.

Architecture:

Frontend → Backend API → PostgreSQL

Cluster:

- 1 Control Plane (master-1)
- 2 Worker Nodes (worker-1, worker-2)
- Flannel CNI
- containerd Runtime

---

## Phase 1: Infrastructure Setup

### Step 1: Create Virtual Machines

**Purpose:** Create machines that will act as Kubernetes nodes.

Created:

- master-1
- worker-1
- worker-2

Using:

```bash
multipass launch --name master-1 --mem 2G --disk 10G
multipass launch --name worker-1 --mem 1G --disk 8G
multipass launch --name worker-2 --mem 1G --disk 8G
multipass list
```

Result:

```
master-1
worker-1

```

### Step 2: Install Kubernetes Components

Installed on all nodes:

- containerd
- kubeadm
- kubelet
- kubectl

**Purpose:**

- `containerd`: runs containers.
- `kubelet`: agent that communicates with the Kubernetes API server and manages pods on the node.
- `kubeadm`: bootstrap and manage the cluster components.
- `kubectl`: CLI to interact with the cluster.

Executed on: `master-1`


- Controller Manager

Result:

### Step 4: Configure kubectl

**Purpose:** Allow `kubectl` to communicate with the new cluster as a non-root user.

Configured user kubeconfig at `~/.kube/config`.

### Step 5: Join Worker Nodes

Executed on: `worker-1`, `worker-2`

Using the `kubeadm join` command printed during init:

```bash
sudo kubeadm join <CONTROL-PLANE-IP>:6443 --token <token> \
  --discovery-token-ca-cert-hash sha256:<hash>
```

**Purpose:** Register workers with the control plane so they can run workloads.

Result: `master-1`, `worker-1`, `worker-2` became cluster members.

## Phase 3: Networking

### Step 6: Install Flannel CNI

**Purpose:** Provide pod-to-pod networking across nodes.

Installed Flannel using the upstream manifest.

---

### Problem Encountered

Flannel pods entered `CrashLoopBackOff`.

Error message indicated `br_netfilter missing` or issues with `/proc/sys/net/bridge`.

**Root Cause:** The worker nodes did not have the required kernel module / sysctl available for bridged networking.

**Fix applied (on each worker):**

```bash
sudo modprobe br_netfilter
echo br_netfilter | sudo tee /etc/modules-load.d/k8s.conf
# ensure required sysctls are set
cat <<EOF | sudo tee /etc/sysctl.d/k8s.conf
net.bridge.bridge-nf-call-iptables = 1
net.ipv4.ip_forward = 1
net.bridge.bridge-nf-call-ip6tables = 1
EOF
sudo sysctl --system
```

Result: Flannel pods started and reached `Running`.

## Phase 4: Application Deployment

Namespace used for the app: `three-tier-app`.

### Step 7: Create Namespace

**Purpose:** Logical isolation for the application resources.

Applied:

```bash
kubectl create namespace three-tier-app
```

---

### Step 8: Create Secret

Created a Kubernetes secret named `postgres-secret` with:

- `POSTGRES_DB`
- `POSTGRES_USER`
- `POSTGRES_PASSWORD`

**Purpose:** Store database credentials securely and inject them into pods.

Example:

```bash
kubectl -n three-tier-app create secret generic postgres-secret \
  --from-literal=POSTGRES_DB=appdb \
  --from-literal=POSTGRES_USER=pguser \
  --from-literal=POSTGRES_PASSWORD=pgpassword
```

---

## Phase 6: Storage

### Step 9: Create Persistent Volume (PV)

Created a PV named `postgres-pv` pointing to host storage `/mnt/data/postgres`.

**Purpose:** Provide physical storage for Postgres data.

Example PV size: `1Gi`.

---

### Step 10: Create Persistent Volume Claim (PVC)

Created `postgres-pvc` in the `three-tier-app` namespace which requests storage from available PVs.


**Purpose:** Let pods request durable storage.

Relationship:

```
Application -> PVC -> PV -> Disk
```

### Problem Encountered

PVC stayed `Pending`. PV was in `Released` state and still tied to `default/postgres-pvc`.

**Fix:** Deleted the problematic PV and recreated it so it would bind to the new PVC.

```bash
kubectl delete pv postgres-pv
# recreate PV with correct claimRef or without claimRef so it can bind
```

Result: PVC became `Bound` and PV `Bound`.

## Phase 7: PostgreSQL Deployment

### Step 11: Deploy PostgreSQL

Created:

- `postgres` Deployment (mounts `postgres-pvc` at `/var/lib/postgresql/data`)
- `postgres` Service (ClusterIP)

Uses secret for credentials and PVC for data.

**Purpose:** Run a persistent Postgres instance for the application.

Postgres flow summary:

```
Postgres Pod -> PVC -> PV -> Host Storage
```

Result: Postgres pod reached `Running`.

## Phase 8: Backend Deployment

### Step 12: Deploy Backend API

Created:

- Backend Deployment
- Backend Service (ClusterIP)

Backend env config:

```
DB_HOST=postgres
DB_PORT=5432
```

**Purpose:** Provide API endpoints that the frontend calls; backend connects to Postgres using service DNS.

### Problem Encountered

Backend pod entered `ErrImageNeverPull`.

**Cause:** The container image used by the deployment existed only locally on `master-1` but the pod was scheduled on `worker-2`, which could not access the local image.

**Learning:** Kubernetes will schedule pods where resources are available; image must be accessible to all nodes (use a registry) or preload the image on each node.

**Fix:** Push the image to Docker Hub and change deployment to pull from the public registry.

```text
docker push parii99/three-tier-backend:v1
# in deployment.yaml set image: parii99/three-tier-backend:v1
# set imagePullPolicy: Always
```

Result: Worker nodes pulled the image from Docker Hub and the backend pod reached `Running`.

### Step 13: Deploy Frontend

- Frontend Service (NodePort)

Image used: `parii99/three-tier-frontend:v1`

Service Type: `NodePort`, port `30080`.

Result: Frontend pod became `Running` and was reachable at `http://<NODE-IP>:30080`.

## Phase 10: Service Discovery (concept)

**Important concept learned:**

Services provide DNS names in-cluster. The backend uses the service name `postgres` instead of a fixed IP. This enables portability and rescheduling without changing configuration.

Flow:

```
Backend -> postgres (service DNS) -> Postgres Service -> Postgres Pod
```

---

## Phase 11: Final Application Flow

End-to-end flow:

```
Browser -> NodePort 30080 -> Frontend Service -> Frontend Pod -> API Service -> Backend Pod -> Postgres Service -> Postgres Pod -> PVC -> PV -> Host Storage
```

---

## Internal Kubernetes Flow Learned

When running `kubectl apply -f app.yaml`, the control flow is:

```
kubectl -> API Server -> etcd -> Scheduler -> select Node -> Kubelet -> containerd -> pull image -> create container -> pod running
```
