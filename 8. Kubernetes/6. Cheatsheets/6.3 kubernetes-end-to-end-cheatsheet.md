# Kubernetes End to End Cheat Sheet
> **From Basics to 3-Tier Application Deployment**

---

## 1. Kubernetes Architecture

```
Control Plane (Master Node)          Worker Nodes
┌──────────────────────────┐         ┌──────────────────────────┐
│  kube-apiserver          │         │  Worker Node 1            │
│  etcd                    │         │  ┌────────┐  ┌─────────┐ │
│  scheduler               │         │  │kubelet │  │kube-proxy│ │
│  controller-manager      │         │  └────────┘  └─────────┘ │
│  cloud-controller-manager│         │        Pods               │
└──────────────────────────┘         └──────────────────────────┘
```

- **Control Plane**: Brain of the cluster. Makes global decisions (scheduling, etc.)
- **Worker Nodes**: Run your applications (containers inside Pods)

---

## 2. Control Plane Components

| Component | Role |
|---|---|
| `kube-apiserver` | Front door of the cluster. All requests go through it. |
| `etcd` | Distributed key-value store. Stores all cluster data. |
| `scheduler` | Selects the best node for new Pods. |
| `controller-manager` | Runs controllers that maintain the desired state. |
| `cloud-controller-manager` | Handles cloud provider integrations (optional). |

---

## 3. How Kubernetes Works (The Big Picture)

```
1. You define desired state (YAML)
        ↓
2. API Server validates & stores the state
        ↓
3. Scheduler decides where to run Pods
        ↓
4. Kubelet creates & runs Pods on nodes
        ↓
5. Kube-proxy enables networking & load balancing
        ↓
6. Your app is running & reachable
```

## 4. Kubeadm – Create a Cluster

### On Master Node
```bash
# 1. Install Docker & dependencies
# 2. kubeadm init
sudo kubeadm init

# 3. Configure kubectl
mkdir -p $HOME/.kube
sudo cp -i /etc/kubernetes/admin.conf $HOME/.kube/config
sudo chown $(id -u):$(id -g) $HOME/.kube/config

# 4. Install a CNI Plugin (Network)
# Example: Flannel, Calico, Weave
kubectl apply -f https://raw.githubusercontent.com/coreos/flannel/master/Documentation/kube-flannel.yml
```

### On Worker Node(s)
```bash
# 1. Install Docker & dependencies
# 2. Get Join Command from Master (--print-join-command)
# 3. Run the join command
kubeadm join <MasterIP>:6443 \
  --token <token> \
  --discovery-token-ca-cert-hash sha256:<hash>
```

## 5. Worker Node Components – Details

| Component | Role |
|---|---|
| `kubelet` | Agent that runs on each node. Ensures containers are running in a Pod. |
| `kube-proxy` | Maintains network rules on the node. Enables Service networking & load balancing. |
| `Container Runtime` | Runs containers. (containerd, Docker, CRI-O) |

### Pod (Smallest Deployable Unit)
```
Pod
├── Container 1 (App)
└── Container 2 (Helper)

- Shared Network (IP)
- Shared Storage (Volumes)
```
> Typically one container per Pod (best practice).

---

## 6. Pod – The Smallest Unit

A Pod represents a running process in your cluster.

**Pod Characteristics:**
- Has its own IP address
- Shares storage (volumes)
- Shares network namespace
- Ephemeral (can be created and destroyed)

> If a Pod dies, a new Pod with a new IP is created.

---

## 7. Deployment – Manage Pods

A Deployment manages ReplicaSets which ensure the desired number of Pods are running.

```
Deployment
    └── ReplicaSet (v1) ──update──> ReplicaSet (v2)
            └── Pods                     └── Pods (New Version)
```

**Features:**
- Declarative updates (Rolling Update)
- Rollbacks
- Self-healing (if a Pod fails, it creates a new one)
- Scaling (increase/decrease replicas)

```bash
kubectl create deployment myapp --image=nginx --replicas=3
kubectl scale deployment myapp --replicas=5
kubectl rollout undo deployment myapp
kubectl rollout status deployment myapp
```

## 8. Service – Networking & Discovery

Services expose Pods via a stable IP/DNS name.

| Service Type | Description | Access |
|---|---|---|
| `ClusterIP` (default) | Internal access inside cluster | Virtual IP (e.g. 10.96.0.10) |
| `NodePort` | Access from outside via `<NodeIP>:<Port>` | External |
| `LoadBalancer` | Maps to external DNS name | External (cloud) |

```
Service (ClusterIP) 10.96.0.10
        ├── Pod 1  10.244.1.2
        ├── Pod 2  10.244.1.3
        └── Pod 3  10.244.1.4

kube-proxy on each node ensures traffic reaches the right Pod.
```

---

## 9. Endpoints – Behind the Scenes

Endpoints are the actual Pod IPs selected by a Service.

- When a Pod is created, its IP is added to Endpoints.
- When a Pod dies, its IP is removed.
- Service always routes to healthy Pods.

```bash
kubectl get endpoints
kubectl describe endpoints <service-name>
```

## 10. Ingress – HTTP/HTTPS Routing

Ingress manages external access (Layer 7).

```
User → Ingress Controller (NGINX Ingress)
            ├── app.com        → frontend-service
            └── api.app.com    → backend-service
```

**Features:**
- Handles host-based routing
- Handles path-based routing
- SSL termination

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-ingress
spec:
  rules:
  - host: app.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend-service
            port:
              number: 80
  - host: api.app.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: backend-service
            port:
              number: 3000
```

## 11. ConfigMap & Secret

| | ConfigMap | Secret |
|---|---|---|
| **For** | Non-sensitive data | Sensitive data |
| **Example Keys** | APP_ENV, APP_PORT, LOG_LEVEL | DB_USER, DB_PASS, API_KEY |
| **Values** | Plain text | Base64 encoded |
| **Used as** | Environment variable / Volume (file) | Environment variable / Volume (file) |

```yaml
# ConfigMap example
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  APP_ENV: production
  APP_PORT: "8080"

# Secret example
apiVersion: v1
kind: Secret
metadata:
  name: app-secret
type: Opaque
data:
  DB_USER: dXNlcg==      # base64
  DB_PASS: cGFzc3dvcmQ=  # base64
```

## 12. Storage – PV & PVC

```
User (PVC Claim) → PVC → PV (Actual Storage)
```

| Resource | Role |
|---|---|
| `PV` (PersistentVolume) | PV provides storage |
| `PVC` (PersistentVolumeClaim) | PVC requests storage |

They are bound together.

**Storage Types:**
- Local Disk
- NFS
- Cloud Disk (EBS, PD)
- Ceph, GlusterFS, etc.

```bash
kubectl get pv
kubectl get pvc
kubectl describe pvc <name>
```

## 14. 3-Tier Application – Real World Example

```
Frontend (Web) → Backend (API) → Database (PostgreSQL)

Ingress (app.com)
    ↓
Frontend Deployment (Port: 80)
    ↓  Service: frontend-svc (ClusterIP)
Backend API Deployment (Port: 3000)
    ↓  Service: backend-svc (ClusterIP)
PostgreSQL StatefulSet (Port: 5432)
    └── PVC (Data)
```

## 18. Troubleshooting Order

```
Node → Pod → Deployment → Service → Endpoints → Ingress → Application
```

| Step | Command |
|---|---|
| Check Node status | `kubectl get nodes` |
| Check Pod status | `kubectl get pods -o wide` |
| Describe a Pod | `kubectl describe pod <name>` |
| Check Pod logs | `kubectl logs <name>` |
| Check Service | `kubectl describe svc <name>` |
| Check Endpoints | `kubectl get endpoints <name>` |
| Check Ingress | `kubectl describe ingress <name>` |
| Check Events | `kubectl get events --sort-by='.lastTimestamp'` |
