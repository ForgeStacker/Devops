# Kubernetes (Kubeadm) Cheat Sheet

> **Goal:** Understand how Kubernetes works as one complete story, from cluster setup to exposing an app in the browser.

This note is written for a beginner who wants to connect the ideas behind `kubeadm`, `kubelet`, `kubectl`, Pods, Deployments, Services, Flannel, NodePort, and Ingress.

## Big Picture

The final goal is to reach your application from a browser.

```text
Browser
	│
	▼
Application
```

In Kubernetes, several layers sit in between.

```text
Browser -> Ingress / NodePort -> Service -> Endpoints -> Pods -> Container -> Application
```

## Why Kubernetes Needs A Cluster

One machine is risky.

If that machine dies:

```text
Application Down
```

So Kubernetes spreads work across multiple machines called **nodes**.

```text
Master Node
Worker Node 1
Worker Node 2
```

## Main Kubernetes Components

| Component | What It Does | Why It Matters |
|---|---|---|
| `kubeadm` | Bootstraps the cluster. | Used to create the control plane and join nodes. |
| `kubelet` | Runs on every node and manages pods. | Keeps containers healthy and running. |
| `kubectl` | Command-line client for Kubernetes. | Lets humans talk to the cluster. |
| `containerd` | Container runtime. | Pulls images and runs containers. |

## kubeadm

`kubeadm` is the tool that creates the Kubernetes cluster.

Example:

```bash
sudo kubeadm init
```

This creates the control plane components:

```text
API Server
Scheduler
Controller Manager
etcd
```

### Why kubeadm Exists

Without `kubeadm`, cluster setup would be manual and much harder.

## kubelet

`kubelet` runs on every node.

It watches the node and makes sure the containers described by Kubernetes are actually running.

```text
Master / Worker Node
		  │
		  ▼
		kubelet
		  │
		  ▼
	  Container
```

### Why kubelet Matters

If a pod should be running and is not, `kubelet` helps bring it back.

## kubectl

`kubectl` is the command-line tool you use to inspect and control the cluster.

Example:

```bash
kubectl get nodes
```

```text
You
 │
 ▼
kubectl
 │
 ▼
API Server
```

## Create The Cluster

Typical flow:

1. Install `kubeadm`, `kubelet`, `kubectl`, and `containerd` on every node.
2. Initialize the master node.
3. Join worker nodes.
4. Install a pod network like Flannel.

### Initialize The Master Node

```bash
sudo kubeadm init --pod-network-cidr=10.244.0.0/16
```

This starts the control plane and prepares the cluster.

### Configure kubectl

Run these commands on the master node so your shell can talk to the cluster:

```bash
mkdir -p $HOME/.kube

sudo cp -i /etc/kubernetes/admin.conf $HOME/.kube/config

sudo chown $(id -u):$(id -g) $HOME/.kube/config
```

Without this, commands like `kubectl get nodes` may fail.

### Join Worker Nodes

The master node gives you a `kubeadm join ...` command.

Run that command on each worker node.

```text
Master
 │
 ├── Worker-1
 └── Worker-2
```

Check cluster nodes:

```bash
kubectl get nodes
```

## Pod Networking With Flannel

Pods need networking so they can talk to each other.

Flannel provides that network.

```text
Pod A
  │
  ▼
Flannel Network
  │
  ▼
Pod B
```

Install Flannel:

```bash
kubectl apply -f flannel.yaml
```

### Common Problem

If Flannel fails, pods may stay in:

```text
ContainerCreating
```

You may also see errors like:

```text
subnet.env not found
```

That usually means the network plugin is not healthy yet.

## Deployment

A Deployment manages pods for you.

Create one:

```bash
kubectl create deployment nginx --image=nginx
```

Visual:

```text
Deployment
	  │
	  ▼
	 Pod
```

### Why Deployment Matters

If a pod dies, the Deployment creates a new one.

### Check Deployments And Pods

```bash
kubectl get deployments
kubectl get pods
```

## Pod Vs Deployment

### Pod

A Pod is the actual running unit.

```text
Pod
 └── NGINX
```

### Deployment

A Deployment manages Pods and keeps the desired number running.

```text
Deployment
		│
		▼
	  Pod
```

## Scaling

Scaling means running more copies of the same app.

Example:

```bash
kubectl scale deployment nginx --replicas=3
```

Visual:

```text
Deployment
		│
 ┌────┼────┐
 ▼    ▼    ▼
Pod1 Pod2 Pod3
```

## Service

Pods change, but users need a stable way to reach them.

A Service solves that.

```text
			 Service
				 │
		┌──────┼──────┐
		▼      ▼      ▼
	 Pod1   Pod2   Pod3
```

Create a Service from a Deployment:

```bash
kubectl expose deployment nginx --port=80
```

Check Services:

```bash
kubectl get svc
```

## ClusterIP

`ClusterIP` is the default Service type.

It gives the Service an internal IP such as:

```text
10.96.166.219
```

### Why It Exists

It is used for communication inside the cluster only.

```text
Service
 │
 ▼
10.96.166.219
```

## Endpoints

Endpoints are the actual Pod IPs behind a Service.

Check them with:

```bash
kubectl get endpoints
```

Example:

```text
10.244.1.2
10.244.2.3
10.244.2.4
```

### Why Endpoints Matter

The Service needs to know which Pods should receive traffic.

```text
Service
 │
 ▼
Endpoints
 │
 ├── Pod1
 ├── Pod2
 └── Pod3
```

## NodePort

ClusterIP is internal only, so the browser cannot use it directly.

NodePort opens a port on every node so external traffic can enter.

Create one:

```bash
kubectl expose deployment nginx \
--type=NodePort \
--port=80 \
--name=nginx-nodeport
```

Example output:

```text
80:32469/TCP
```

Flow:

```text
Browser
	│
	▼
NodeIP:32469
	│
	▼
Service
	│
	▼
Pods
```

## Ingress

NodePort works, but it becomes awkward when you have many apps.

Example:

```text
Frontend
Backend
Payment
```

If each one uses a different NodePort, managing them becomes messy.

Ingress gives you a cleaner entry point.

```text
Browser
	│
	▼
Ingress
	│
	├── Frontend
	├── Backend
	└── Payment
```

## Ingress Controller

Ingress rules alone do nothing.

You also need an Ingress Controller to read those rules and route traffic.

Install the NGINX Ingress Controller:

```bash
kubectl apply -f https://raw.githubusercontent.com/kubernetes/ingress-nginx/main/deploy/static/provider/cloud/deploy.yaml
```

Flow:

```text
Ingress Rule
		│
		▼
Ingress Controller
		│
		▼
Service
```

## Ingress Rule

Example rule:

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress

spec:
  ingressClassName: nginx

  rules:
  - http:
		paths:
		- path: /
		  pathType: Prefix
		  backend:
			 service:
				name: nginx
				port:
				  number: 80
```

Apply it:

```bash
kubectl apply -f ingress.yaml
```

Check it:

```bash
kubectl get ingress
```

## Useful Commands

| Command | What It Does |
|---|---|
| `kubectl get nodes` | Show cluster nodes. |
| `kubectl get pods` | Show pods. |
| `kubectl get deployments` | Show deployments. |
| `kubectl get svc` | Show services. |
| `kubectl get ingress` | Show ingress resources. |
| `kubectl get endpoints` | Show endpoints behind services. |
| `kubectl describe pod <pod>` | Show detailed pod information. |
| `kubectl logs <pod>` | Show pod logs. |
| `kubectl create deployment nginx --image=nginx` | Create a deployment from an image. |
| `kubectl scale deployment nginx --replicas=3` | Scale the deployment. |
| `kubectl expose deployment nginx --port=80` | Create a Service for a deployment. |
| `kubectl expose deployment nginx --type=NodePort --port=80 --name=nginx-nodeport` | Expose the app outside the cluster. |
| `kubectl delete pod <pod>` | Delete a pod. |
| `kubectl apply -f file.yaml` | Create or update resources from a manifest. |
| `kubectl delete -f file.yaml` | Delete resources defined in a manifest. |

## Common Beginner Problems

| Problem | Likely Cause | What To Check |
|---|---|---|
| `kubectl get nodes` does not work | kubectl is not configured | Check `$HOME/.kube/config`. |
| Pods stay in `ContainerCreating` | Network plugin is not ready | Check Flannel pods and CNI setup. |
| Service is created but browser cannot open it | You used `ClusterIP` | Use `NodePort` or `Ingress`. |
| Ingress rules do not route traffic | Ingress Controller is missing | Install the controller first. |
