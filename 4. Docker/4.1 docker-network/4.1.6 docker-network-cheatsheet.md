## Introduction

A **Docker Network** is a virtual network that connects containers to each other, to the host machine, and to the outside world.

By default, containers are **isolated** — they cannot talk to each other unless you put them in the same network.

Docker networking solves three problems:
1. How do containers talk to **each other**?
2. How does the **host** access a container?
3. How do containers reach the **internet**?

> **Open `4.3.1`** — it shows all three answers visually with network diagrams.

## Part 1 — Network Types (Foundation)

> **Image `4.3.1`** covers all 5 network types with diagrams.
> **Image `4.3.3`** has a compact driver comparison table for quick reference.

### All Network Drivers at a Glance

| Driver | Scope | Isolation | Use Case |
|--------|-------|-----------|----------|
| `bridge` | Single host | Good | Default for standalone containers |
| `user-defined bridge` | Single host | Better | Multi-container apps — **recommended** |
| `host` | Single host | None | High performance, monitoring tools |
| `none` | Single host | Complete | Fully isolated, no network needed |
| `overlay` | Multi-host (Swarm) | Good | Connect containers across multiple hosts |
| `macvlan` | Single host | High | Give container its own MAC on LAN |

---

### Bridge (Default)

> **`4.3.1` — Section A** shows this network type.

The default network when you use `docker run` without specifying anything.

- Docker creates a virtual bridge called `docker0`
- Containers get IPs from subnet `172.17.0.0/16`
- Containers can reach the internet via NAT
- ⚠️ **No DNS** — containers cannot find each other by name, only by IP

```bash
docker run -d --name myapp nginx          # lands on default bridge automatically
docker run -p 8080:80 nginx               # host port 8080 → container port 80
```

**⚠️ Problem with default bridge:** No DNS resolution between containers.
If backend tries to connect to `db:3306` it will fail because `db` cannot be resolved.
This is what causes the issue shown in **`4.3.2`** — read that next.

---

### User-Defined Bridge (Recommended)

> **`4.3.1` — Section B** and **`4.3.3` — Section 1** show this.
> **`4.3.4`** shows a full real app using this pattern.

You create this yourself. It is the correct way for any multi-container app.

- **Built-in DNS** — containers find each other by container name (not IP)
- Better isolation than default bridge
- Each compose project gets its own user-defined bridge automatically

```bash
docker network create my-network
docker run --network my-network --name web nginx
docker run --network my-network --name app node:alpine
# "app" can now reach "web" at http://web:80 — by name, no IP needed
```

✅ This is what **`4.3.4`** uses — 3 separate named networks for proper isolation.

---

### Host Network

> **`4.3.3` — Section 2** shows this.

Container shares the host's network stack — no isolation, no NAT.

- Container uses the host's IP and ports directly
- No port mapping needed
- Best for: monitoring agents, high-performance tools
- ⚠️ Not for production multi-container apps — no isolation

```bash
docker run --network host nginx            # nginx binds directly to host port 80
```

---

### None Network

> **`4.3.3` — Section 3** shows this.

Zero network access — fully isolated.

```bash
docker run --network none alpine           # no internet, no container communication
```

---

### Overlay Network

> **`4.3.1` — Section E** and **`4.3.3` — Section 4** show this.

Connects containers across **multiple Docker hosts** (Docker Swarm / Kubernetes).
Uses VXLAN encryption. Not needed for single-host setups.

```bash
docker network create -d overlay my-overlay
```

## Part 2 — The Problem: Mixed Setup (docker run + docker compose)

> **⚠️ Open `4.3.2` now** — it shows exactly this scenario with network diagrams and error messages.
> **`4.3.2` is the problem. The solutions are below.**

This is the **most common beginner mistake** with Docker networking.

### What Happens

You start Frontend and Backend with `docker run`, and Database with `docker compose`.

```bash
# docker run (App containers)
docker run -d --name frontend -p 8080:80 nginx
docker run -d --name backend -p 5000:5000 node:18-alpine node server.js

# docker-compose.yml (Database)
services:
  db:
    image: mysql:8
```

Result — they land on **different networks**:

| Container | Method | Network | Subnet |
|-----------|--------|---------|--------|
| Frontend (Nginx) | `docker run` | `bridge` (default) | `172.17.0.0/16` |
| Backend (Node.js) | `docker run` | `bridge` (default) | `172.17.0.0/16` |
| Database (MySQL) | `docker compose` | `myapp_default` | `172.18.0.0/16` |

> **`4.3.2`** shows this split visually — two separate network boxes with a broken connection between them.

### Why It Breaks

- `docker run` → default bridge (`172.17.0.0/16`)
- `docker compose` → creates its own project network (`myapp_default`, `172.18.0.0/16`)
- These two bridge networks are **isolated from each other**
- Backend tries `db:3306` → DNS lookup for `db` fails → **connection refused**

Errors you will see (shown in **`4.3.2`**):
```
getaddrinfo ENOTFOUND db
connect ECONNREFUSED 172.18.0.2:3306
ETIMEDOUT
```

---

### 3 Fixes (All Shown in `4.3.2`)

**Fix 1 — Best: Put everything in docker compose** ✅

```yaml
# docker-compose.yml
services:
  frontend:
    image: nginx
  backend:
    image: node:18-alpine
    command: node server.js
  db:
    image: mysql:8
```

All three join `myapp_default` automatically → backend reaches `db:3306` by name ✅

---

**Fix 2 — Connect the docker run container to the compose network**

```bash
docker network connect myapp_default backend
# backend now joins myapp_default and can reach db by name or IP
```

---

**Fix 3 — Use host network (not for production)**

```bash
docker run --name backend --network host node:18-alpine node server.js
# both share host network — removes isolation entirely
```

## Part 3 — The Solution: 3-Tier App with Proper Isolation

> **Open `4.3.4`** — it shows the full architecture with 3 separate networks.
> **Open `4.3.5`** alongside it — it shows exactly which ports are exposed vs internal.
> This is the **correct answer** to the problem shown in `4.3.2`.

### Why 3 Separate Networks?

The problem in `4.3.2` used 0 named networks — everything was accidental.
The solution in `4.3.4` uses 3 intentional named networks:

| Network | Subnet | Who's On It | Purpose |
|---------|--------|-------------|---------|
| `frontend_net` | `172.18.0.0/16` | Nginx + App | Web tier can talk to app tier |
| `backend_net` | `172.19.0.0/16` | App + MySQL | App tier can talk to data tier |
| `data_net` | `172.20.0.0/16` | MySQL only | DB isolated — extra protection |

### Who Can Talk to Who

> **`4.3.4` — Network Architecture View** (top right of image) shows this exactly.

| From → To | Can Communicate? | Why |
|-----------|-----------------|-----|
| Browser → Nginx | ✅ | Port 8080 published to host |
| Nginx → App | ✅ | Both on `frontend_net` |
| App → MySQL | ✅ | Both on `backend_net` |
| Nginx → MySQL | ❌ | No shared network |
| Internet → MySQL | ❌ | DB never published to host |
| Internet → App | ❌ | App never published to host |

### How a Request Flows

> **`4.3.4` — HOW IT WORKS (FLOW)** section shows this step by step.

```
1. User opens http://<host-ip>:8080
2. Request hits Docker Host on published port 8080
3. Forwarded to Nginx container (frontend_net) on port 80
4. Nginx forwards to App container (backend_net) on port 5000
5. App queries MySQL (backend_net/data_net) on port 3306
6. MySQL returns data → App returns JSON → Nginx returns response → User sees page
```

### docker-compose.yml

> This is the full example shown in **`4.3.4`**.

```yaml
version: '3.8'

networks:
  frontend_net:
    driver: bridge
  backend_net:
    driver: bridge
  data_net:
    driver: bridge

volumes:
  db_data:

services:
  web:
    image: nginx:alpine
    ports:
      - "8080:80"              # ← only published port — entry point from internet
    networks:
      - frontend_net
    depends_on:
      - app

  app:
    image: node:18-alpine
    working_dir: /usr/src/app
    volumes:
      - ./:/usr/src/app
    command: node server.js
    expose:
      - "5000"                 # ← internal only — not reachable from host
    networks:
      - frontend_net           # ← can talk to web
      - backend_net            # ← can talk to db
    depends_on:
      - db

  db:
    image: mysql:8
    environment:
      MYSQL_ROOT_PASSWORD: root
      MYSQL_DATABASE: appdb
    volumes:
      - db_data:/var/lib/mysql
    expose:
      - "3306"                 # ← internal only — never exposed to internet
    networks:
      - backend_net            # ← app can reach db here
      - data_net               # ← extra isolation layer
```

## Part 4 — Port Mapping

> **Open `4.3.5`** — it shows exactly which ports are public vs internal across the 3 tiers.

```
Host port 8080  →  Container (Nginx) port 80    ← published, anyone can access
                   Nginx → App port 5000         ← internal, same Docker network only
                   App → DB port 3306            ← internal, same Docker network only
```

| In docker-compose.yml | Meaning |
|-----------------------|---------|
| `ports: - "8080:80"` | Published — host port 8080 maps to container port 80. Accessible from outside. |
| `expose: - "5000"` | Internal only — other containers on same network can reach it. Host cannot. |

```bash
# Port mapping format
docker run -p <host-port>:<container-port> image

docker run -p 8080:80 nginx        # host:8080 → container:80
docker run -p 5000:5000 node-app   # host:5000 → container:5000
```

## Part 5 — Container Communication Rules

> **`4.3.3` — Container Communication Rules** section shows this as a visual checklist.

| Scenario | Result |
|----------|--------|
| Same network | ✅ Can communicate — by container name or IP |
| Different networks | ❌ Cannot communicate — isolated by default |
| Container → Host | ✅ Via published ports or gateway IP |
| Container → Internet | ✅ Via NAT (outbound only) |

## Part 6 — Best Practices

> **`4.3.3` — Best Practices** section confirms all of these.

- ✅ Use **user-defined bridge networks** — never rely on the default bridge
- ✅ Use **docker compose for all services** that need to talk to each other — never mix with bare `docker run`
- ✅ Give networks **meaningful names** (`frontend_net`, `backend_net`, `data_net`)
- ✅ Use `expose` for internal ports, `ports` only for the public entry point
- ✅ Keep the **database on a private network only** — never publish DB port to host in production
- ✅ Use **overlay** only for multi-host setups (Docker Swarm)
- ✅ Design with least privilege — if two containers don't need to talk, don't put them on the same network

## Quick Reference Commands

> **`4.3.3` — Common Commands** table shows all of these.

```bash
# Networks
docker network ls                                        # list all networks
docker network inspect <network>                         # inspect (see subnet, containers)
docker network create my-net                             # create bridge network
docker network rm my-net                                 # remove network
docker network connect my-net <container>                # add container to network
docker network disconnect my-net <container>             # remove container from network

# Run with network
docker run --network my-net nginx                        # join specific network
docker run --network host nginx                          # use host network
docker run --network none alpine                         # no network

# Port mapping
docker run -p 8080:80 nginx                              # host:8080 → container:80

# Testing (from inside a container)
docker exec -it <container> wget -qO- http://app:5000/health    # test HTTP to another container
docker exec -it <app> mysql -h db -u root -proot appdb          # test DB connection
docker exec -it <container> nslookup db                         # test DNS resolution
```
