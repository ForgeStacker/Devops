# CI/CD Pipeline — Jenkins + Docker + Container Registry

## What is CI/CD?

**CI (Continuous Integration)**

Automatically:
- Pull latest code
- Build application
- Run tests (optional)
- Create artifacts (JAR, Docker image, etc.)

**CD (Continuous Delivery/Deployment)**

Automatically:
- Push artifacts/images to a registry
- Deploy to target environments (VM, Kubernetes, Cloud Run, etc.)

```
CI  =  Build & Verify
CD  =  Release & Deploy
```

> **This cheatsheet covers CI only** — it stops after pushing the image to the registry.
> Deployment (VM, Kubernetes, Cloud Run, ECS, etc.) belongs to the CD stage and is covered in a separate cheatsheet.

## The Full Flow (No Cloud / No VM)

```
1. Developer writes code and runs: git push origin main
2. GitHub receives the push and fires a Webhook (HTTP POST) to Jenkins
3. Jenkins receives the webhook and starts the pipeline automatically
4. Jenkins checks out the latest code from GitHub
5. Jenkins builds the application (install deps, compile — if needed)
6. Jenkins builds a Docker image from the Dockerfile
7. Jenkins pushes the Docker image to a Container Registry (GAR / Docker Hub)
8. Image is now stored and versioned — ready for any deployment target
```

## Components

| Component | Role |
|-----------|------|
| **GitHub** | Stores source code. Sends webhook on push. |
| **Webhook** | HTTP POST request from GitHub → Jenkins to trigger pipeline |
| **Jenkins** | CI/CD automation server. Runs pipeline as per Jenkinsfile. |
| **Docker** | Builds container image using Dockerfile |
| **Container Registry** | Central image store (GAR / Docker Hub / ECR) |

## Part 1 — Webhook Setup

Jenkins must be reachable from GitHub for webhooks to work.

### How it works
1. Webhook is configured in GitHub with your Jenkins URL:
   ```
   http://<jenkins-url>/github-webhook/
   ```
2. On every `git push`, GitHub sends a POST request to Jenkins
3. Jenkins receives it and starts the pipeline

### GitHub → Repository Settings → Webhooks → Add Webhook
- **Payload URL:** `http://<jenkins-url>/github-webhook/`
- **Content type:** `application/json`
- **Trigger:** Just the push event

## Part 2 — Jenkinsfile

The `Jenkinsfile` lives in the root of your repository. Jenkins reads it and runs each stage.

### Basic Structure

```groovy
pipeline {
    agent any

    environment {
        IMAGE = "your-registry/your-repo/my-app:latest"
    }

    stages {

        stage('Checkout Code') {
            steps {
                git 'https://github.com/user/repo.git'
            }
        }

        stage('Build Application') {
            steps {
                sh 'npm install'
                sh 'npm run build'
            }
        }

        stage('Build Docker Image') {
            steps {
                sh 'docker build -t ${IMAGE} .'
            }
        }

        stage('Push to Registry') {
            steps {
                sh 'docker push ${IMAGE}'
            }
        }

    }
}
```

### Stages Explained

| Stage | What Happens |
|-------|-------------|
| **Checkout Code** | Jenkins pulls latest code from GitHub repo |
| **Build Application** | Install dependencies, compile, run tests (optional — skip if not needed) |
| **Build Docker Image** | Creates Docker image from Dockerfile, tags it |
| **Push to Registry** | Pushes image to Container Registry — image is now stored and versioned |

## Part 3 — Dockerfile (What Jenkins Builds)

```dockerfile
# Example Node.js app
FROM node:18-alpine
WORKDIR /usr/src/app
COPY package*.json ./
RUN npm install
COPY . .
EXPOSE 3000
CMD ["node", "server.js"]
```

Jenkins runs `docker build -t my-app:latest .` using this file.

### Docker Build Command Explained

```bash
docker build -t my-app:latest .
```

| Part | Meaning |
|------|---------|
| `docker build` | Build an image from a Dockerfile |
| `-t` | Tag the image with a name |
| `my-app:latest` | Image name : tag |
| `.` | Build context — current directory (where the Dockerfile lives) |

## Docker Image vs Container

| | Docker Image | Docker Container |
|--|--|--|
| **What it is** | Blueprint / template | Running instance of an image |
| **Stored in** | Container Registry | Docker host (memory/disk) |
| **Created by** | `docker build` | `docker run` |
| **State** | Static file | Running / Stopped |

```
Dockerfile  →  docker build  →  Image  →  docker run  →  Container
                                  ↓
                             docker push
                                  ↓
                           Container Registry
```

> The CI pipeline produces the **Image**.
> The **Container** is created later during deployment (CD stage).

## Part 4 — Container Registry

After the pipeline runs, the image is stored in a registry.

### Before Pushing — Authenticate

Jenkins must log in to the registry before it can push images.

```bash
# Non-interactive login — use this in Jenkinsfile
echo $PASS | docker login -u $USER --password-stdin

# Interactive login — local testing only
docker login
```

In the Jenkinsfile, credentials come from Jenkins Credentials store (see Part 5).

### Why a Registry?

- Central storage for Docker images
- Share images between different environments
- Versioning (`v1`, `v2`, `latest`)
- Makes deployment consistent and reliable

### Common Registries

| Registry | Provider | Free Tier |
|----------|----------|-----------|
| **Docker Hub** | Docker | Yes (public) |
| **GAR** (Google Artifact Registry) | Google Cloud | Yes |
| **ECR** (Elastic Container Registry) | AWS | Yes |
| **GHCR** (GitHub Container Registry) | GitHub | Yes |

### Image Tag Format

```bash
# Docker Hub
docker.io/username/my-app:latest

# Google Artifact Registry
us-central1-docker.pkg.dev/project/repo/my-app:latest

# GitHub Container Registry
ghcr.io/username/my-app:latest
```

## Part 5 — Jenkins Credentials

Jenkins stores secrets securely — never hardcode them in Jenkinsfile.

### Credentials You Need

| Credential | Type | Purpose |
|------------|------|---------|
| `github-pat` | Username + Password | Jenkins clones private GitHub repo |
| `registry-key` | Secret file / Username+Password | Jenkins pushes to Container Registry |

### Using Credentials in Jenkinsfile

```groovy
stage('Checkout Code') {
    steps {
        git credentialsId: 'github-pat',
            url: 'https://github.com/user/repo.git'
    }
}

stage('Push to Registry') {
    steps {
        withCredentials([usernamePassword(
            credentialsId: 'registry-creds',
            usernameVariable: 'USER',
            passwordVariable: 'PASS'
        )]) {
            sh 'echo $PASS | docker login -u $USER --password-stdin'
            sh 'docker push ${IMAGE}'
        }
    }
}
```

### Adding Credentials in Jenkins UI
1. Jenkins Dashboard → Manage Jenkins → Credentials
2. Add Credential → choose type
3. Give it an ID (e.g. `github-pat`) — use this ID in Jenkinsfile

## Part 6 — How Jenkins Retrieves the Jenkinsfile

> Based on Image 4 from your notes.

When your pipeline is configured to use SCM (Source Control Management):

```
1. Script calls Jenkins API for pipeline config.xml
       ↓
2. Gets: Repo URL + Credential ID + Jenkinsfile path
   (Script does NOT get the actual token — Jenkins keeps it secure)
       ↓
3. Script creates a temporary helper freestyle job in Jenkins
       ↓
4. Helper job: checkout repo using credentialId, copy Jenkinsfile, archive it
       ↓
5. Jenkins clones repo using its saved credential internally
       ↓
6. Script downloads the archived Jenkinsfile artifact
       ↓
7. Jenkinsfile saved locally for scanning/analysis
```

**Key point:** Your script never sees the actual GitHub token. Jenkins uses it internally.

## Common Problems

### Webhook Not Triggering
- Jenkins URL is not reachable from GitHub
- Webhook URL is wrong — must end with `/github-webhook/`
- Check: GitHub → Repository → Settings → Webhooks → Recent Deliveries

### Jenkins Cannot Clone Repository
- `github-pat` credential missing or incorrect
- Repository is private and no `credentialsId` set in the `git` step

### Docker Build Fails
- `Dockerfile` does not exist in the workspace root
- Docker daemon not running on the Jenkins server
- Wrong build context path

### Docker Push Fails
- Missing `docker login` step before push
- Wrong registry path or image tag
- Registry credentials not configured in Jenkins

### Image Not Visible in Registry
- Wrong repository path used in the image tag
- Push succeeded but you're checking the wrong tag

## Quick Reference — Common Jenkins Commands

```bash
# Trigger pipeline manually via API
curl -X POST http://<jenkins-url>/job/<job-name>/build \
  --user user:api-token

# Check pipeline status
curl http://<jenkins-url>/job/<job-name>/lastBuild/api/json

# Docker commands Jenkins runs internally
docker build -t my-app:latest .          # build image
docker push my-app:latest                # push to registry
docker images                            # list images
docker image prune -f                    # clean up old images after push
```

## Summary

```
GitHub        →  stores code, sends webhook on push
Webhook       →  notifies Jenkins automatically
Jenkins       →  reads Jenkinsfile, runs stages
Jenkinsfile   →  defines: checkout → build app → build image → push
Docker        →  builds the image from Dockerfile
Registry      →  stores the image, versioned and ready
```

> The pipeline ends at the registry push.
> What happens after (deploy to a server, VM, Kubernetes) is a separate concern.
