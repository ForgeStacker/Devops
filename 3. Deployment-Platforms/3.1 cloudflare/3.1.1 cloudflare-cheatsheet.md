# Cloudflare — Complete Cheatsheet

> Covers: Cloudflare Pages (frontend deployment) + Cloudflare Workers (backend/full stack) + Cloudflare D1 (database)

---

## Introduction

**Cloudflare** is a global edge network platform. It has 200+ data centers around the world. When someone visits your website, Cloudflare serves it from the location closest to them — making it fast everywhere.

Cloudflare offers two main ways to deploy your app:

| | Cloudflare Pages | Cloudflare Workers |
|---|---|---|
| **What it is** | Static frontend hosting with auto-deploy from GitHub | Serverless JavaScript runtime at the edge |
| **Best for** | React, Vite, HTML/CSS/JS frontend apps | API backends, full stack apps, dynamic logic |
| **Has database?** | No | Yes — via Cloudflare D1 (SQLite) |
| **Deploy method** | Push to GitHub → auto-build and deploy | `wrangler deploy` via CLI |
| **Custom domain?** | Yes — with DNS + HTTPS auto-setup | Yes |
| **Pricing** | Generous free tier | Generous free tier |

**When to use what:**
- Building a **frontend-only** site (React, Vite)? → Use **Cloudflare Pages**
- Building a **full stack app** (frontend + API + database)? → Use **Cloudflare Workers + D1**
- Need **both**? → You can combine them, or use Workers to serve static assets too

## Wrangler — Language Support

Wrangler is **not language-agnostic**. It is built specifically for the **Cloudflare Workers runtime**, which has its own rules about what it can run.

### Supported Languages

| Language | How It Works | Notes |
|----------|-------------|-------|
| **JavaScript** | Native — runs directly | Best supported, no extra setup |
| **TypeScript** | Native — wrangler compiles it automatically | Recommended for larger projects |
| **Rust** | Compiled to WebAssembly (Wasm) | Supported via `workers-rs` crate |
| **C / C++** | Compiled to WebAssembly (Wasm) | Possible but uncommon |
| **Python** | Beta support (as of 2024) | Limited — runs via Pyodide in Wasm |

### NOT Supported (Cannot Use with Wrangler)

| Language | Why Not Supported |
|----------|-------------------|
| **Go** | Does not compile to a Workers-compatible Wasm target cleanly |
| **Java** | JVM does not run in the Workers runtime |
| **PHP** | Not supported in the edge runtime |
| **Ruby** | Not supported |
| **Node.js APIs** | Workers is NOT Node.js — many Node built-ins (`fs`, `net`, `child_process`) are unavailable |

> **Important:** Cloudflare Workers does NOT run Node.js. It uses its own V8-based runtime. You cannot use `require()` (CommonJS) in older Workers — use ES modules (`import/export`) instead. Some Node.js built-ins are partially polyfilled via the `nodejs_compat` compatibility flag.

### Enabling Node.js Compatibility (Partial)

Add this to `wrangler.toml` to enable partial Node.js API support:

```toml
compatibility_flags = ["nodejs_compat"]
```

This gives access to some Node.js APIs like `Buffer`, `crypto`, `path` — but not all.

## Part 1 — Cloudflare Pages (Frontend Deployment)

### What Is Cloudflare Pages?

Cloudflare Pages is a hosting platform for static websites and frontend apps.

You push your code to GitHub → Cloudflare automatically builds it → deploys it globally.

Every time you push new code, a new version goes live automatically. No manual steps needed.

---

### Step 1: Develop on Your Laptop

Create a Vite + React project and test it locally before deploying anything.

```bash
npm create vite@latest my-app    # create a new Vite project
cd my-app
npm install                      # install dependencies
npm run dev                      # start local dev server
```

Your app runs at `http://localhost:5173` — only on your machine while the dev server is running.

When done testing:
```bash
Ctrl + C                         # stop the dev server
```

---

### Step 2: Push Code to GitHub

GitHub stores your code in the cloud. Cloudflare Pages connects to GitHub and watches for changes.

```bash
git init                                    # initialize git in your project
git add .                                   # stage all files
git commit -m "Initial commit"              # save a snapshot

# Go to github.com → create a new repository (e.g. my-app)
# Copy the repo URL, then:

git remote add origin <github-repo-url>    # connect local repo to GitHub
git push -u origin main                    # push code to GitHub
```

Your code is now safely stored on GitHub.

---

### Step 3: Connect Cloudflare Pages to GitHub (Auto Deploy)

Cloudflare Pages watches your GitHub repo. Every push triggers an automatic build and deploy.

**Setup steps (done once in Cloudflare dashboard):**

1. Go to [dash.cloudflare.com](https://dash.cloudflare.com) → **Pages**
2. Click **Create a project** → **Connect to Git**
3. Select your GitHub repository
4. Select root directory (or project subfolder)
5. Set build settings:

| Setting | Value |
|---------|-------|
| Framework | Vite |
| Build command | `npm run build` |
| Build output directory | `dist` |

6. Click **Deploy**

Cloudflare will:
- Clone your repo
- Run `npm install`
- Run `npm run build`
- Generate static files in `dist/`
- Deploy them to its global CDN

Your site goes live at: `https://<project-name>.pages.dev`

---

### Step 4: What Happens on Every Push

After the one-time setup, this is the full cycle for every code change:

```
You edit code on laptop
  ↓
git add . && git commit -m "update" && git push
  ↓
GitHub receives new changes
  ↓
Cloudflare Pages detects the push
  ↓
Auto build starts (npm install + npm run build)
  ↓
New version deploys to global CDN
  ↓
Site is live with your changes — automatically
```

No manual deploy step needed. Just push to GitHub.

---

### Step 5: Add a Custom Domain

By default your site is at `something.pages.dev`. To use your own domain (e.g. `practice.yourdomain.com`):

**Buy a domain** from a registrar (e.g. Hostinger, Namecheap, Google Domains).

**Add domain to Cloudflare:**
1. Go to Cloudflare dashboard → **Pages** → your project
2. Click **Custom Domains** → **Add a domain**
3. Enter your domain (e.g. `practice.yourdomain.com`)
4. Cloudflare will tell you to add a DNS record

**Create a CNAME record in Cloudflare DNS:**

| Type | Name | Content | Proxy | TTL |
|------|------|---------|-------|-----|
| CNAME | practice | `<your-project>.pages.dev` | Proxied | Auto |

**Wait for DNS propagation** — takes a few minutes to a few hours.

Cloudflare automatically verifies the domain and issues an SSL certificate.

---

### Step 6: HTTPS / SSL (Automatic)

Once your custom domain is connected:

- Cloudflare handles SSL automatically (via Let's Encrypt)
- HTTPS is enforced — all traffic is secured
- No manual certificate setup needed

Your site is now live at `https://practice.yourdomain.com` with:
- Automatic HTTPS
- Fast global CDN
- Auto-deploy on every push

## Part 2 — Cloudflare Workers (Full Stack App)

### What Is Cloudflare Workers?

A **Cloudflare Worker** is a serverless JavaScript function that runs at the edge — on Cloudflare's network, close to your users.

It can:
- Serve your frontend static files
- Handle API requests (GET, POST, PUT, DELETE)
- Talk to a database (Cloudflare D1)
- Run custom backend logic

Think of it as a tiny server that lives everywhere at once — no VPS, no nginx, no scaling to manage.

---

### Full Stack Architecture

```
User / Browser
    │
    │  1. HTTPS Request
    ▼
Cloudflare Worker  (runs at nearest edge location)
    │
    ├── Request for /         → serves index.html, CSS, JS  (static frontend)
    │
    └── Request for /api/*    → runs backend logic
              │
              │  SQL over HTTPS
              ▼
        Cloudflare D1  (SQLite database)
              │
              │  Returns rows
              ▼
        Worker sends JSON response
              │
              ▼
        Frontend updates UI
```

---

### Step 1: Local Development

Install wrangler (Cloudflare's CLI tool):

```bash
npm install -g wrangler
wrangler login                  # authenticate with your Cloudflare account
```

Project structure:
```
my-app/
├── src/
│   └── index.js                # Worker code (backend logic)
├── web/                        # Frontend Vite app
│   └── dist/                   # Built static files (after npm run build)
├── migrations/
│   └── 0001_init.sql           # SQL to create DB tables
└── wrangler.toml               # Configuration file
```

Start local development:
```bash
wrangler dev                    # runs Worker locally + uses local D1 (.wrangler/state/)
```

---

### Step 2: Create and Configure Cloudflare Resources

```bash
wrangler d1 create expense-db   # creates a D1 database in Cloudflare
                                # outputs a database_id — copy it
```

Update `wrangler.toml`:

```toml
name = "my-app"
main = "src/index.js"
compatibility_date = "2024-01-01"

[[d1_databases]]
binding = "DB"                  # how you reference the DB in your Worker code
database_name = "expense-db"
database_id = "<paste-your-database-id-here>"

[assets]
directory = "../web/dist"       # where your built frontend files are
```

---

### Step 3: Set Up the Database (Migrations)

Write your table schema in `migrations/0001_init.sql`:

```sql
CREATE TABLE expenses (
  id      INTEGER PRIMARY KEY AUTOINCREMENT,
  title   TEXT NOT NULL,
  amount  INTEGER NOT NULL,
  date    TEXT NOT NULL
);
```

Apply migrations to the cloud database:

```bash
wrangler d1 migrations apply expense-db --remote
# Creates the tables in your Cloudflare D1 database
```

---

### Step 4: Write the Worker (Backend Logic)

The Worker is a JavaScript file that handles all requests:

```js
// src/index.js

export default {
  async fetch(request, env) {
    const url = new URL(request.url);

    // API: GET all expenses
    if (request.method === "GET" && url.pathname === "/api/expenses") {
      const { results } = await env.DB.prepare("SELECT * FROM expenses").all();
      return Response.json(results);
    }

    // API: POST new expense
    if (request.method === "POST" && url.pathname === "/api/expenses") {
      const { title, amount, date } = await request.json();
      await env.DB.prepare(
        "INSERT INTO expenses (title, amount, date) VALUES (?, ?, ?)"
      ).bind(title, amount, date).run();
      return Response.json({ success: true });
    }

    // Serve frontend static files for all other routes
    return env.ASSETS.fetch(request);
  }
};
```

**API Endpoints:**

| Method | Route | Action |
|--------|-------|--------|
| GET | `/api/expenses` | Fetch all expenses |
| POST | `/api/expenses` | Add a new expense |
| PUT | `/api/expenses/:id` | Update expense by ID |
| DELETE | `/api/expenses/:id` | Delete expense by ID |

---

### Step 5: Build the Frontend

```bash
cd web
npm run build                   # compiles Vite app → outputs files to web/dist/
```

The `dist/` folder contains your ready-to-serve HTML, CSS, and JS files.

---

### Step 6: Deploy Everything

```bash
wrangler deploy
```

This single command uploads:
- Your Worker JavaScript (backend)
- Static files from `web/dist/` (frontend)
- D1 database binding configuration

Your full stack app is live at:
```
https://<your-worker-name>.workers.dev
```

## Part 3 — Key Differences Side by Side

### Pages vs Workers — When to Use Each

| Scenario | Use |
|----------|-----|
| Portfolio site, landing page, blog | Cloudflare Pages |
| React/Vite app with no backend | Cloudflare Pages |
| App that needs an API | Cloudflare Workers |
| App that needs a database | Cloudflare Workers + D1 |
| Auto-deploy from GitHub pushes | Cloudflare Pages |
| Full control over request handling | Cloudflare Workers |

### Deployment Comparison

| Step | Cloudflare Pages | Cloudflare Workers |
|------|------------------|--------------------|
| Write code | VS Code | VS Code |
| Local dev | `npm run dev` | `wrangler dev` |
| Deploy trigger | `git push` → GitHub → auto | `wrangler deploy` |
| Build step | Cloudflare runs `npm run build` | You run `npm run build` first |
| Config file | Dashboard (GUI) | `wrangler.toml` |
| Live URL | `<name>.pages.dev` | `<name>.workers.dev` |
| Custom domain | Yes (CNAME + DNS) | Yes |
| Database | No | Yes — D1 (SQLite) |

## Part 4 — Complete Flow: Code to Live

### Cloudflare Pages Flow

```
Write Code (VS Code)
  → npm run dev           (test locally on localhost:5173)
  → git push              (send to GitHub)
  → Cloudflare detects push
  → Auto: npm install + npm run build
  → Deploy dist/ to CDN
  → Live on <name>.pages.dev
  → Add CNAME in DNS
  → Live on your custom domain with HTTPS
```

### Cloudflare Workers Flow

```
Write Code (VS Code)
  → wrangler dev          (test locally with local D1)
  → wrangler login        (authenticate)
  → wrangler d1 create    (create cloud database)
  → write migrations SQL
  → wrangler d1 migrations apply --remote    (create tables)
  → npm run build         (build frontend)
  → wrangler deploy       (push everything to Cloudflare)
  → Live on <name>.workers.dev
```
