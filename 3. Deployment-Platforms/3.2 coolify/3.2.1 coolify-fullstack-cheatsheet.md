# Coolify Full Stack Deployment — Cheatsheet

---

## Introduction

**Coolify** is a self-hosted platform that lets you deploy full stack applications on your own server — the same way Heroku or Render work, but on a server you own and control.

You push code to GitHub. Coolify detects it, pulls the code to your server, and uses **Docker** to build and run your app automatically.

The full stack runs inside **Docker containers** on your server:
- A **PostgreSQL database** container
- A **Node.js backend** container
- A **Nginx frontend** container

**Traefik** acts as the reverse proxy — it sits in front of all containers and routes incoming requests to the right one based on the domain name.

**Cloudflare DNS** points your domain to your server's IP.

```
User visits coolify.yourdomain.com
  → Cloudflare DNS resolves domain to your Server IP
  → Request hits your server on port 80
  → Traefik checks the Host header
  → Routes to the correct container
  → Container sends back the response
```

---

## Aims — What We Want to Achieve

1. **Self-host a full stack app** on your own server (Google Cloud, AWS, any VPS)
2. **Automate deployments** — push to GitHub → Coolify deploys automatically
3. **Run all services in Docker** — database, backend, frontend each in their own container
4. **Route traffic correctly** — Traefik sends requests to the right container by domain
5. **Connect domain to server** — Cloudflare DNS points your domain to your server IP
6. **Enable API communication** — frontend talks to backend, backend talks to database

## How to Achieve It — Step by Step

---

### Step 1: Push Code to GitHub

Coolify needs your code on GitHub to pull and deploy it.

```bash
git add .
git commit -m "update"
git push
```

GitHub stores your code (the repo). Coolify will clone from here.

---

### Step 2: Install Coolify on Your Server

Coolify runs on your own server (e.g. Google Cloud VM, DigitalOcean, any VPS).

SSH into your server and install Coolify:

```bash
curl -fsSL https://cdn.coollabs.io/coolify/install.sh | bash
```

Access Coolify dashboard at: `http://<your-server-ip>:8000`

Coolify handles:
- Pulling code from GitHub
- Building and running Docker containers
- Managing deployments, logs, environment variables, secrets

---

### Step 3: Connect GitHub to Coolify

In the Coolify dashboard:

1. Go to **Sources** → **Add GitHub Account**
2. Authorize Coolify to access your GitHub repositories
3. Coolify gets access via SSH key or Personal Access Token

Now Coolify can clone your repo onto the server automatically.

---

### Step 4: Coolify Clones the Repository

When a deployment is triggered, Coolify runs:

```bash
git clone https://github.com/<user>/<repo>.git
```

Your code now exists on the server inside Coolify's working directory:

```
full-stack-deploy/
├── backend/
│   └── Dockerfile
├── web/
│   └── Dockerfile
└── docker-compose.yml
```

---

### Step 5: Coolify Reads docker-compose.yml

Coolify reads your `docker-compose.yml` to understand:
- What services to run
- Which ports to expose
- What volumes (persistent data) to create
- What networks to connect
- What labels to apply (for Traefik routing)

Example `docker-compose.yml`:

```yaml
version: "3.8"

networks:
  app-network:
    driver: bridge

volumes:
  postgres-data:

services:

  db:
    image: postgres:16
    environment:
      POSTGRES_USER: admin
      POSTGRES_PASSWORD: secret
      POSTGRES_DB: myapp
    ports:
      - "5432:5432"
    volumes:
      - postgres-data:/var/lib/postgresql/data
    networks:
      - app-network

  api:
    build: ./backend
    ports:
      - "3000:3000"
    environment:
      DATABASE_URL: postgres://admin:secret@db:5432/myapp
    depends_on:
      - db
    networks:
      - app-network

  web:
    build: ./web
    ports:
      - "80:80"
    depends_on:
      - api
    networks:
      - app-network
    labels:
      - "traefik.enable=true"
      - "traefik.http.routers.expense.rule=Host(`coolify.yourdomain.com`)"
      - "traefik.http.services.expense.loadbalancer.server.port=80"
```

---

### Step 6: Docker Builds Images

For each service that has a `Dockerfile`, Docker builds an image.

An **image** is like a blueprint — it contains your app code, runtime, and all dependencies packaged together.

**Example Dockerfile for the backend (Node.js):**
```dockerfile
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
EXPOSE 3000
CMD ["node", "index.js"]
```

**Example Dockerfile for the frontend (Nginx):**
```dockerfile
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build

FROM nginx:alpine
COPY --from=builder /app/dist /usr/share/nginx/html
EXPOSE 80
```

Docker builds each image:
```bash
docker build -t backend-image ./backend
docker build -t frontend-image ./web
```

---

### Step 7: Docker Starts Containers

Docker creates and starts a **container** from each image.

A **container** is a running instance of an image — it is isolated, lightweight, and has its own process space.

```bash
docker compose up -d        # start all containers in background
```

What's running inside your server:

| Container | Image | Internal Port | Role |
|-----------|-------|---------------|------|
| `db` | postgres:16 | 5432 | Stores all database data |
| `api` | built from ./backend | 3000 | Runs Node.js server |
| `web` | built from ./web | 80 | Nginx serves static files |
| `traefik` | traefik | 80 (external) | Reverse proxy, routes by domain |

All containers are in the **same Docker network** (`app-network`).  
They communicate using **service names** as hostnames:
- `api` container connects to db at `db:5432`
- `web` container calls backend at `api:3000`

---

### Step 8: Traefik Routes Incoming Requests

**Traefik** is a reverse proxy that runs as a container. It:
- Listens on port 80 (HTTP) and 443 (HTTPS)
- Reads **labels** from other containers
- Matches the `Host` header of incoming requests
- Forwards the request to the correct container

Example labels on the `web` container:
```yaml
labels:
  - "traefik.enable=true"
  - "traefik.http.routers.expense.rule=Host(`coolify.yourdomain.com`)"
  - "traefik.http.services.expense.loadbalancer.server.port=80"
```

This tells Traefik:
- Enable routing for this container
- If the request Host is `coolify.yourdomain.com` → route to this container
- Forward to port 80 inside the container

---

### Step 9: Point Your Domain to the Server

**Buy a domain** from a registrar (Hostinger, Namecheap, etc.).

**Add domain to Cloudflare**, then create a DNS record:

| Type | Name | Content | Proxy |
|------|------|---------|-------|
| A | coolify | `34.131.4.141` (your server IP) | Proxied or DNS only |

Now `coolify.yourdomain.com` resolves to your server's IP.

## How a Real Request Works (End to End)

### Website Request (Frontend)

```
1. User opens coolify.yourdomain.com in browser

2. Cloudflare DNS
   → domain: coolify.yourdomain.com
   → resolves to Server IP: 34.131.4.141

3. Request arrives at server on port 80 (HTTP)

4. Traefik Reverse Proxy receives it
   → checks Host header: coolify.yourdomain.com
   → matches label rule on web container
   → forwards request to web container port 80

5. Nginx (web container)
   → serves React app (index.html + JS/CSS assets)

6. Browser renders the app
   → User sees your application
```

### API Request (Frontend → Backend → Database)

```
1. User action triggers API call in browser
   (e.g. fetch('/api/expenses') or axios.get('/api/expenses'))

2. Frontend sends HTTP request to API container
   → API container is at api:3000 inside Docker network

3. Node.js (API container) receives request
   → runs query against database

4. PostgreSQL (db container) processes query
   → returns result rows

5. API container sends JSON response back to frontend

6. Frontend updates the UI with the data
```

## Key Concepts

| Term | What It Means |
|------|---------------|
| **Coolify** | Self-hosted deployment platform — like Heroku but on your own server |
| **Docker Image** | Blueprint of your app — built from a Dockerfile |
| **Docker Container** | Running instance of an image — isolated process |
| **docker-compose.yml** | File that defines all your services, their config, and how they connect |
| **Docker Network** | Private network containers share — they talk using service names |
| **Volume** | Persistent storage for a container (e.g. database data survives restarts) |
| **Traefik** | Reverse proxy that reads container labels and routes requests by domain |
| **Reverse Proxy** | Sits in front of your containers, receives all traffic, forwards to correct service |
| **Cloudflare DNS** | Translates your domain name into your server's IP address |

## Summary — What Happens From Push to Live

```
1. You push code to GitHub
2. Coolify detects the push (webhook)
3. Coolify pulls (clones) the repo onto the server
4. Coolify reads docker-compose.yml
5. Docker builds images from each Dockerfile
6. Docker starts containers (db, api, web, traefik)
7. All containers join the same Docker network
8. Traefik reads labels and sets up routing rules
9. Cloudflare DNS points domain to server IP
10. User visits domain → Traefik routes → correct container responds
11. Frontend ↔ Backend ↔ Database (all inside Docker network)
```

## Quick Reference Commands

```bash
# On your server — Docker basics
docker compose up -d              # start all containers (detached)
docker compose down               # stop all containers
docker compose logs -f            # follow logs of all containers
docker compose logs api           # logs of a specific service
docker compose ps                 # list running containers
docker compose restart api        # restart a specific service

# Rebuild after code changes
docker compose up -d --build      # rebuild images and restart

# Access a container shell
docker exec -it <container-name> sh

# Check Traefik routing
docker logs traefik

# Database access
docker exec -it db psql -U admin -d myapp
```

## Why This Stack?

| Benefit | Explanation |
|---------|-------------|
| **Self-hosted** | You own the server and data — no vendor lock-in |
| **Automated deploys** | Push to GitHub → Coolify deploys automatically |
| **Isolated services** | Each part (db, api, web) runs in its own container |
| **Easy scaling** | Spin up more containers of any service |
| **Persistent data** | Docker volumes keep database data safe across restarts |
| **Domain routing** | Traefik routes multiple domains/subdomains to different containers |
| **Free (infra cost only)** | Coolify is open source — you only pay for the server |
